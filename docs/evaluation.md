# Evaluation Replication

This document explains how to reproduce the current MemCoder evaluation from a
fresh checkout, assuming access to the required local model weights,
SHINE checkpoints, CUDA hardware, and OpenAI judge credentials.

## What Is Evaluated

The current evaluation compares a SHINE-LoRA code-QA path against Qwen3-8B
baselines on generated design-doc QA tasks.

Included target repositories:

- `target_repos/antirez__kilo`
- `target_repos/marimo-team__marimo`

Included evaluation artifacts:

- `artifacts/antirez__kilo/easy/docs/` and `artifacts/marimo-team__marimo/easy/docs/`
  contain generated design documents.
- `artifacts/*/easy/qas/` and `artifacts/*/easy/qas_v1/` contain evaluation QA
  pairs.
- `artifacts/*/ledger.json` maps document IDs to docs, QA files, LoRA paths, and
  routing metadata.
- `prompts/llm_judge_grading_v2.md` is the current default judge rubric.
- `schemas/` documents the JSON contracts consumed and emitted by the harness.

Compared systems:

- `naive`: Qwen3-8B with no document context and no LoRA.
- `in_context`: Qwen3-8B with the ground-truth design document placed in prompt
  context.
- `shine`: Qwen3-8B with a SHINE-generated LoRA loaded and no source document in
  the prompt.
- `embedding` routing configs compare oracle LoRA selection against precomputed
  embedding-based retrieval for the `shine` condition.

## One-Time Setup

```bash
git submodule update --init --recursive
uv sync
```

Edit the relevant `config/eval/*.yaml` file so `model.qwen_base` points to the
local Qwen3-8B Hugging Face snapshot and `model.shine_root` points to
`vendor/SHINE`. The existing configs record the paths used on the original run
machine and should be treated as examples.

Set judge credentials:

```bash
export OPENAI_API_KEY=...
```

or place the key in `.env`, matching `judge.dotenv_path` in the config.

## Ensure LoRAs Exist

The `shine` condition needs `.pt` LoRA dictionaries referenced by each
`ledger.json`. If a ledger entry has no LoRA path yet, bake it from the design
document:

```bash
uv run python scripts/generate_shine_lora.py \
  --config config/eval/kilo_easy_v0.yaml \
  --design-doc artifacts/antirez__kilo/easy/docs/overview_purpose_1.json
```

Repeat for every document ID you want included. The script updates the ledger
unless `--no-update-ledger` is provided.

## Run The Main Evaluation

Kilo easy, oracle routing:

```bash
uv run python scripts/run_eval.py all --config config/eval/kilo_easy_v0.yaml
```

Marimo easy, oracle routing:

```bash
uv run python scripts/run_eval.py all --config config/eval/marimo_easy_v0.yaml
```

Each run writes a new directory:

```text
results/<run_name>_<YYYYMMDDTHHMM>/
```

The important outputs are:

- `predictions.jsonl`: raw model outputs and routing metadata.
- `judgments.jsonl`: LLM-as-judge scores, reasoning, and failure modes.
- `report.md`: aggregate scores by condition and document.
- plot files generated by `eval.plots`.
- `run_config.yaml` and `manifest.json`: reproducibility snapshot.

To rerun only part of the pipeline:

```bash
uv run python scripts/run_eval.py predict --config config/eval/kilo_easy_v0.yaml
uv run python scripts/run_eval.py judge --run-dir results/<run_name>_<timestamp>
uv run python scripts/run_eval.py report --run-dir results/<run_name>_<timestamp>
```

## Replicate Embedding Routing

Generate routing JSONL files from a ledger and QA file:

```bash
uv run python scripts/embedding_router.py \
  --lora-store artifacts/antirez__kilo/ledger.json \
  --qa-pairs artifacts/antirez__kilo/easy/qas/overview_purpose_1.json \
  --output artifacts/antirez__kilo/easy/routing_results.qwen3.overview_purpose_1.top1.jsonl \
  --top-k 1
```

Repeat for the QA files listed in
`config/eval/kilo_easy_v0_embedding.yaml`, then run:

```bash
uv run python scripts/run_eval.py all --config config/eval/kilo_easy_v0_embedding.yaml
```

## Reproduce Analysis Notes

The pilot interpretation in `docs/observations.md` was computed from judged run
outputs and rubric/prompt comparison scripts under `scripts/`, especially:

- `scripts/prompt_ab_test.py`
- `scripts/rubric_impact.py`
- `scripts/plot_naive_hard.py`
- `scripts/score_results_jsonl.py`

Those scripts consume `predictions.jsonl` and `judgments.jsonl` style artifacts
from `results/` run directories. If the original run directories are not
present, regenerate them with the configs above before running analysis.

## Notes And Limits

The RAG-over-design-docs condition described in the initial plan is a required
future baseline but is not implemented in the current `scripts/run_eval.py`
condition set. The implemented alternative systems for the current grading
artifact are the `naive` and `in_context` Qwen3-8B baselines plus routing
variants for SHINE.
